# Beat Copilot

指摘への返信に書いた根拠を、PR 本文に書くべき汎用観点として create-pr スキルへ Edit で還元する。同型の指摘が次の PR では「本文を読めば解消済み」になるため。

## 1. 指摘を分類する

対応済みの各 AI 指摘を 3 つに分け、還元対象を絞る:

- **実欠陥**: コードを直して対応したもの。本文では防げない → 対象外
- **本文で潰せた**: diff の外にある前提 (デプロイ単位・到達不能経路・上流での検証・既決着の設計判断) を知らないために出た指摘 → §2 へ
- **本文に書いてあった**: 書いてあったのに出た指摘。落とし込んでも効かない → 対象外

「対応不要」と返信した指摘はほぼ「本文で潰せた」組。返信に書いた根拠こそ、本文に先回りして書くべきだった内容。

## 2. create-pr に落とし込む

1. 返信の根拠を「この PR 固有の事実」から「PR 本文に書くべき観点」へ汎用化する。作業リポ固有の語 (サービス名・フィールド名・ブランチ名) を残さない
2. create-pr の SKILL.md を読み、同等の観点が既にあれば「対応済み」と報告して終える (以降のステップはスキップ)。近い観点があれば新規追加ではなく既存行の拡張を優先する — カタログの肥大は観点の選択精度を下げる
3. Edit は `readlink -f` で解決した実体パスに対して行う (Edit ツールは symlink 越しの書き込みを拒否するため)
4. Edit 後、skill-md-reviewer サブエージェント (`subagent_type: skill-md-reviewer`) に skill-md-guide を基準としたレビューを依頼する。反復回数・収束条件は skill-md-guide のレビュー工程に従う
5. 変更 diff と「どの指摘がどう潰れるようになるか」をユーザに提示する。dotfiles のコミットはユーザ判断
