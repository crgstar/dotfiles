# CLAUDE.md 作成・編集ガイド

## 基本原則

CLAUDE.md は全セッションで context を消費する。行数が増えるほど Claude は個々のルールを軽視し始めるので、**1 行ごとに「これがないと Claude がミスするか？」を問い、No なら削除する**。

## 含めるもの

Claude がコード・lint・標準仕様から *推測できない* 情報だけ:

- 推測できないビルド・テスト・リントコマンド
- 言語のデフォルトと異なるコーディング規約
- ブランチ命名・コミット規約
- 必要な環境変数・特殊なセットアップ手順
- アーキ制約（「API 呼び出しはページコンポーネントからのみ」等）

## 含めないもの

- コードを読めば分かること（技術スタック・ディレクトリ展開）
- 言語/フレームワーク標準
- ESLint / Prettier 等で強制可能なルール
- システムプロンプトに既に書かれている規範（二重指示）
- 他の CLAUDE.md（個人グローバル ↔ プロジェクト）と重複する内容
- 頻繁に変わる情報
- 長い説明（`@docs/file.md` で参照する）

具体例は [references/examples.md](references/examples.md)。

## 書き方

- 箇条書きと見出しで構造化する
- 具体的に: 「2 スペースインデント」 > 「コードを適切にフォーマットする」
- `IMPORTANT:` は乱用しない。2 行連続したら片方を降格 — 並べると重要度が薄れる
- 擬似見出し（`典型パターン:` のような末尾コロン段落）は `###` / `####` に昇格
- 本体は 300 行以下（理想 100 行）。超えたら `@docs/*.md` / `.claude/rules/*.md` に分割
- 行数が制限内でも、*特定タスクでしか必要にならない* specialized detail（hook 仕様 / 内部 API 詳細 / 特定運用フロー等）は `.claude/rules/<topic>.md` に分離

## ファイル階層

チーム共有 (Git 管理) は `CLAUDE.md` とトピック別の `.claude/rules/*.md`。
個人用はプロジェクトの `CLAUDE.local.md` と、全プロジェクト共通の `~/.claude/CLAUDE.md`。

## ワークフロー

### 新規作成

1. プロジェクト構造・ビルド/テスト・規約を把握
2. Claude がコードから推測できない情報だけ抽出
3. 「含める/含めない」「書き方」に沿って書く
4. 既存 docs/ や設定ファイルと重複がないか確認

### 添削/レビュー（「添削」「レビュー」「整理」「圧縮」「audit」）

1. 対象（CLAUDE.md / CLAUDE.local.md / `.claude/rules/*.md`）を Read
2. 各行に「含めないもの」を当てて削除/移動/圧縮候補を抽出
   - **セクション単位でも判定**: そのセクションは毎セッション必要か？特定タスク限定なら `.claude/rules/` 候補
   - **実行手順群の保護**: 一連の実行フローを成す番号付きステップ群は分割・移動候補にしない（実行順序が壊れるため）
3. 候補をリスト化してユーザーに提示
4. **議論余地のある候補は `AskUserQuestion` で確認** — 「明示的に書いた注意喚起」を理由なく削るリスクを避ける
5. ピンポイントは `Edit`、再構成は `Write`

判断に迷ったら [references/examples.md](references/examples.md) と [references/anti-patterns.md](references/anti-patterns.md) を参照。
