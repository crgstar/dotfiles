# CLAUDE.md アンチパターン

## 1. 肥大化した CLAUDE.md

**症状**: 300行超。Claude がルールを無視し始める。
**原因**: あれもこれもと情報を詰め込みすぎ。
**対策**: 各行に「これがないと Claude がミスするか？」を問う。No なら削除。

## 2. スタイルガイドの転記

**症状**: ESLint / Prettier のルールを CLAUDE.md に書いている。
**原因**: リンターで強制できるルールを LLM に任せている。
**対策**: リンターに任せる。CLAUDE.md にはリンターで表現できない規約のみ書く。

## 3. コードから推測可能な情報の記載

**症状**: 技術スタック一覧、依存ライブラリ一覧、ディレクトリの全展開。
**原因**: package.json や tsconfig.json を読めば分かることを書いている。
**対策**: Claude がコードを読んで推測できない情報だけ残す。

## 4. /init の出力をそのまま使用

**症状**: 自動生成のまま未レビュー。不要な情報が混在。
**原因**: `/init` は出発点であり完成品ではない。
**対策**: 生成後に手動で各行をレビューし、不要な行を削除する。

## 5. 詳細情報の埋め込み

**症状**: DB スキーマ、API 仕様、長い手順書が CLAUDE.md に直接書かれている。
**原因**: `@` 参照の活用不足。
**対策**: 詳細は `docs/` や `.claude/rules/` に置き、`@path/to/file` で参照する。

## 6. タスク固有の情報

**症状**: 特定機能の実装方法や一時的なワークアラウンドが書かれている。
**原因**: CLAUDE.md に全セッション共通でない情報を入れている。
**対策**: 全セッションに普遍的に適用される情報のみ残す。

## 7. 内容の重複

**症状**: 同じ情報が CLAUDE.md と docs/ の両方にある。
**原因**: コピーペーストで情報が二重化。
**対策**: 正規の場所に一元化し、CLAUDE.md からは `@` で参照する。
