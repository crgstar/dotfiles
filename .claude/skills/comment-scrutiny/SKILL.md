# Comment Scrutiny

## 判定の軸

**残すのは「消すと壊れる」コメントだけ** — 消したとき将来の編集者が壊す方向に動くもの。それ以外（コードや型から読める情報、定石レベルの why、別システムへの言及、重複）は削除する。残す場合も why の核だけに短縮する。

テストコードの fixture・テストデータ選択の意図コメントは、guard として正当化できても削除に倒す — 意図は値の形・テスト名・assert で表現できるため。

## 短縮時の書き方

抽象語（「到達不能パス」）より失敗時に具体的に何が起きるかを書く。
例: 「submitting=true のまま return し、ボタンが固まる」

why / note / reason などのプレフィックスラベルは付けない。コメントは理由の記述であり、ラベルは冗長。

## 手順

1. 差分からコメントを含む行を抽出
2. 各コメントの周辺コードを読み、「消すと壊れる」かを判定して分類
3. 結果を表形式で報告

## 出力形式

| 対象 | 処理 | 判断理由 |
|---|---|---|
| `file.ts` L42 コメント内容 | 削除 | コードの言い換え。`x ?? default` から読める |
| `file.ts` L58 コメント内容 | 短縮 → `提案文` | 消すと壊れるが挙動説明が冗長 |
| `file.ts` L73 コメント内容 | 残す | spread で narrow 崩れ回避。消すと flat 化される |
