# explain — サブエージェント反復付き HTML ドキュメント作成

このスキルは、コード・ライブラリ・概念・設計などについて「**永続化される説明資料**」を
作る依頼を受けたときの、執筆〜レビュー反復の標準フローをまとめたものです。
今ここで雑に説明するだけで十分な質問には使いません。ユーザが「ドキュメント化したい」
「HTML で残したい」「.local/ にまとめて」と明示してきたときの仕事のやり方です。

## 何を作るか

- **出力先**: 作業対象リポジトリの `.local/` 配下に HTML を配置する。
  ファイル数は内容に応じて柔軟に決める（1 枚で済むなら 1 枚、テーマが独立しているなら
  分割。サブエージェントレビューでの分割提案が出たら従う）
- **構成**: タイトル + 左サイドバー TOC + 本文セクション + フッタが基本だが、
  内容に合わせて崩していい（例: ランディングだけリッチに、参考資料は質素に等）
- **スタイル**: `<style>` ブロックに CSS 埋め込みを既定とする（外部 CSS 不要で 1 ファイル
  完結のほうが運用が楽）。ただし内容上必要なら CSS を書き換えてよい。
  たとえば配色を変える・新しいコンポーネント用のクラスを足す・印刷向けスタイルを
  追加するのは全部 OK。**触ってはいけないルールではない**
- **図解**: Mermaid を CDN（jsdelivr）から読み込むのが既定。必要なら静的 SVG / 画像 /
  別の図ライブラリでも構わない
- **読み手のデフォルト**: ユーザ本人（エンジニア）。Go・一般的な開発ツール・
  基本的な SaaS の名前は通じる前提で書く

スキャフォールド（CSS 込みの HTML 骨格）が `assets/template.html` にある。
出発点としては便利だが、テーマ・色・レイアウトを自由に変えてよい。
**雛形を絶対視せず、扱う題材に合った見せ方を優先する**こと。

### 複数ファイルになったとき

題材が複数のテーマに分かれる（例: 設計と全体像 / ライブラリ入門 / 運用ガイドが
それぞれ独立して読める）場合、無理に 1 枚にせず複数 HTML に分けてよい。
分けた場合は次の点に気をつける：

- 各 HTML のサイドバー末尾に「関連: 別ドキュメントへのリンク」ブロックを置く
- 相互参照（A から B への前方リンク、B から A への逆リンク）を必要な箇所で張る
- 重複する解説は片方に集約し、もう一方からはリンクで参照する

## ワークフロー

### 1. 着手前ヒアリング（大規模ドキュメントは必須）

「HTML が 2 枚以上になりそう」「図解を含む」「複数の独立したテーマがある」のいずれかに当てはまれば大規模と判断し、以下の論点を AskUserQuestion で合意する。
ただしユーザが章立て・粒度・図解をすでに明示指定していれば合意済みとしてスキップする。

- **読み手** — デフォルトはユーザ本人。社内同僚・社外公開・将来の自分（メモ）に
  振るときだけ AskUserQuestion で確認
- **章立て** — 概要 → 設計 → 機能 のトップダウンか、ユースケース起点か、
  「設計思想」を厚くするか。複数案を並べて選ばせる。
  コード仕様（型・シグネチャ）・未確定事項・検討経緯は除外がデフォルト。含める場合は選択肢として問う — 問われていない章は後から削除要求が来る典型
- **粒度** — パッケージ/型/責務まで踏み込むか、ファイル単位の責務まで止めるか、
  コマンドライン仕様だけにするか。「メソッド名・変数名には踏み込まない」のような
  スコープ宣言は最初にとっておくと、レビュー時に粒度違反指摘が暴れない
- **図解** — Mermaid 中心 / インライン SVG 中心 / 図最小

なぜ必須か: 書いてから章立てや粒度を直すのは高コスト。3〜4 問の事前ヒアリングで
合意しておくと、後の手戻りが激減し、サブエージェントレビューも論点が安定する。

### 2. 一次情報を取る（勘で埋めない）

書く前に裏取りを終わらせる。

- **ライブラリ・SDK・API・CLI ツール・クラウドサービス** に触れる場合は
  `context7`（`resolve-library-id` → `query-docs`）で公式ドキュメントを引く。
  除外条件は context7 の MCP server instructions に従う
- **コードベース** はソースを直接読む（Read / Grep / Glob）。型・interface 名は
  正確に引用する
- **過去の決定** は `git log` / コミットメッセージから引く。「なぜこの形か」の
  根拠が要る箇所は特定コミットを `git show` で本文まで読み、推測で書かない
- **既存ドキュメント** がリポジトリ内にある場合は重複を避け、相互参照する

「たぶんこう」「一般にこう」で書くと、レビュー時に必ず一次情報との突き合わせで
指摘が立つ。書く前に確認、を徹底する。

### 3. 執筆

`assets/template.html` をコピーしてベースを作るのが手っ取り早いが、絶対ではない。
題材に合わせて以下のような判断は自由：

- 標準テンプレでうまく表現できないなら CSS を書き換える・新クラスを足す・色味を変える
- 単純な索引ページなら TOC を省く
- ランディング的に見せたい場合、ヒーローセクションを大きく取る
- 文章の補助として大きめのインフォグラフィックを置きたい場合、CSS Grid / Flexbox の
  レイアウトを別途組む

ただし、以下は「品質を担保するために多くの場合守ったほうがよい」緩めの規約：

- 章番号と TOC の番号を一致させる（テンプレの CSS counter を使うと自動）
- 各セクションは `<section id="...">` を持ち、TOC のアンカーリンクと対応させる
- コードブロックは `<pre><code>` で、Go の型表記など特殊記号を含むときは HTML エスケープ
- コメントは「why」を優先（HTML 内コメントも）。「何をしているか」はマークアップ
  自体が示すので書かない

ユーザが好む対照スタイル（題材に合えば積極的に使う）：
- 「世間一般の使い方」 ↔ 「このリポジトリでの使われ方」を 2 カラムで並べる `.pair`
- 「やる/やらない」「短所/長所」の callout
- 用語集を最後に置いて、本文の社内/業界用語を引けるようにする

CSS や構成の変更が大きくなりすぎたら、レビューでサブエージェントに「本文の質を
妨げていないか」を併せて見てもらう（観点に「装飾過多になっていないか」を追加する）。

### 4. 図解は Mermaid 中心 — メタ文字に注意

Mermaid 10 系では `flowchart` のノードラベルに以下の記号を含めるとパーサが落ちる
ことが多い。**実コードの型表記をそのままラベルに使わない**のが安全。
`sequenceDiagram` は寛容だが、Note over の `<br/>` を多用すると崩れる場合がある。

| 記号 | 失敗しやすい例 | 安全な書き換え |
|------|----------------|----------------|
| `[ ]` | `args[1]` | `args の 2 要素目` |
| `{ }` | `map[string]interface{}` | `動的データ map` |
| `[/...内に記号/]` | `[/"args[1]"/]` | 通常の `[...]` 矩形にする |
| ラベル先頭の記号 | `[/-flag/]` | 文字で説明する |

複雑な依存関係を 1 図に詰めず、図を分けたほうが結果的に読み手に伝わる。
ノード数が 10 を超えたら分割を検討する。

#### レンダリングを必ず実機で確認する

Mermaid は仕様（ダブルクォートで囲めば OK）と実装挙動が一致しない領域があり、
コードレビューだけでは描画エラーを事前判定できない。Mermaid を含む HTML は
生成後に必ず実機でレンダリングを確認する。手段の優先順位：

1. `mmdc`（`@mermaid-js/mermaid-cli`）が利用可能なら、生成した Mermaid ブロックを
   `mmdc -i in.mmd -o out.svg` で検証する
2. live preview MCP（`veelenga/claude-mermaid` 等）が利用可能なら、それを使う
3. どちらも無ければ、`open` で HTML を開いた後、ユーザに「Mermaid 図が描画されて
   いるか見てください」と頼む。`open` 自体は Claude 側に結果を返さないので、
   完了確認の代替にはならない

レンダリング確認はサブエージェントレビュー（§5）の **前** に済ませる。
描画エラーがあると後段のレビューが無駄になる。

### 5. サブエージェント反復レビュー（このスキルの肝）

ドラフトが書けたら **必ず** `doc-reviewer` サブエージェント（`subagent_type: doc-reviewer`）を
呼んでレビューさせる。レビューを省くと事実誤認・章立てのねじれが残ったまま手渡されるので、
ここを省略する判断はしない。

#### 1 回目のレビュー依頼テンプレート

サブエージェントは会話文脈を持たないので、以下を必ず渡す：

1. **レビュー対象** — HTML のフルパス
2. **参考ファイル** — 突き合わせるべきコード・設定・既存ドキュメントのパス
3. **合意済みの前提** — 読み手・章立て・粒度・図解形式。`§1` で合意したものをそのまま列挙
4. **この SKILL.md のフルパス** — doc-reviewer が §5 の観点・重要度ラベル・形式制約を Read する
5. **レビュー観点** — 事実誤認 / 重要事項の欠落 / 設計思想の捉え方 / 粒度違反 /
   図解の妥当性（Mermaid のラベルや矢印を含む） / 章立て・流れ / 用語集の網羅性
6. **重要度ラベル** — `[critical] / [major] / [minor] / [nit]` で分類するよう指示
7. **形式制約** — 1500 字以内・末尾に `総括: 指摘ありN件 / 指摘なし` の 1 行を必須化
8. **不要なこと** — 「テスト書け」「コード変えろ」「設計に異議あり」など、
   HTML ドキュメント自体の改善でない提案は対象外と明記

#### 2 回目以降のレビュー依頼テンプレート

- 前回の指摘とその対応を箇条書きで明示
- 「**新規発見だけ報告**。既に解消済みのものを列挙する必要は無し」と明示
- 指摘ゼロなら 1〜2 行で完了するよう指示
- 「nit は読み手の理解を本当に妨げるものだけ」と絞らせる

#### 反復の終了条件

次のいずれかで終了：

1. サブエージェントが `総括: 指摘なし` を返す
2. **反復回数の上限に達する**（デフォルト 1 回 — 1 回で主要な指摘はほぼ取れる。ユーザの指示で増やせる）

上限到達時は反復を打ち切り、残指摘を重要度別に整理してユーザに提示し、
「対応 / 見送り / 別タスク化 / さらに N 回反復」を選んでもらう。

反復中に critical が連続して出るときは、§2 の裏取りが甘い可能性があるので、
上限とは別に反復を止めて一次情報の確認に戻る。

### 6. 完了報告

ユーザへの最終報告には以下を含める：

- 作成物のフルパス
- 章立て（番号付き）
- 反映した主要な指摘（critical / major のみ。nit は省略可）
- 関連ドキュメントが複数ある場合は相互リンクの方向
- **上限に達して打ち切った場合**は、未対応の指摘一覧（重要度別）と、それぞれに対する
  ユーザの選択肢（対応 / 見送り / 別タスク化 / 反復追加）

## やってはいけないこと

- **レビュー反復をスキップしてユーザに渡す** — 「素早く渡したい」誘惑が出るが、
  最低 1 回はサブエージェントレビューを通す
- **一次情報を読まずに書く** — 「たぶんこうだろう」で書くと必ずレビューで揺り戻る
- **Mermaid に Go の型表記やシェルのフラグをそのまま埋める** — §4 のメタ文字落とし穴
- **`open` を完了確認の代替にする** — `open` は GUI 起動だけで Claude 側に結果は
  返らない。Mermaid 描画 OK か等はユーザに頼むか、別の検証経路を通す（§4 末尾）
- **AskUserQuestion を省略して大規模ドキュメントを書き始める** — 章立ての作り直しが
  最も高コスト
- **本文を書かずに CSS の装飾だけで仕上げる** — テンプレを変えること自体は自由だが、
  装飾に時間を取られて本文・図解・対照表の質が薄くなるのは本末転倒

## 適用範囲外

- 短い口頭説明で済む質問（「Xって何？」「この行どういう意味？」）— その場で答える
- 設計メモ・ADR・RFC・意思決定の記録 — 形式と読み手が違う
- README や CONTRIBUTING など、リポジトリのトップに置く公式ドキュメント — `.local/` ではなく
  リポジトリ直下に置くべきで、誰でも読む前提なので章立てが変わる
