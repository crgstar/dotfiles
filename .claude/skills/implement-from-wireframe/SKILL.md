## 着手前に UI ライブラリの構造的制約を確認する

WF を読み込んだ時点で、実装方式と使用 UI ライブラリの内部構造を照合する。
why: 構造的に非互換なまま着手すると、スタイリング方法の試行錯誤が複数回発生する。

非互換が判明したら、ハックのリスクと代替方針をセットでユーザに委ねる。

## コメントに外部成果物を参照させない

コメントは「WF を持っていない読者」が読む前提で書く。
why: コードを読む人がそのワイヤーフレームにアクセスできるとは限らない。

外部成果物由来の値でも、値が成立する操作上・仕様上の根拠だけをコメントに書く。コメントの数値がコード中と重複していないかも確認する。
