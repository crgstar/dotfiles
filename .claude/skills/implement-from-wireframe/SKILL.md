# Implement from Wireframe

ワイヤーフレームは「見た目の意図」しか持たない。データ制約・既存パターン・配置先の文脈は別途取りに行く必要があるのが本スキルの前提。

## 最低限のタスク分割 (順序ミスで取りこぼしが起きやすいため)

特に「配置先の調査」「データ仕様の特定」「テスト観点の合意」「最後の突き合わせ」は飛ばしがち。着手前に TodoWrite で以下を並べる:

1. ワイヤーフレームの構造化抽出 (具体値を表に書き出し + 欠落要素の明示)
2. データ仕様の特定 (情報源の確定 + フィールド制約の抽出)
3. 参考実装の特定 (ユーザ確認 + Explore 並列で候補出し)
4. 参考実装の精読 (本体・配置先・兄弟・共通基盤・テストを全部)
5. UI ライブラリ構造的制約の確認 (WF が要求する実装方式と非互換でないか)
6. 設計合意 (変更ファイル / トレードオフ / テスト観点)
7. 実装委譲 (TDD スキルへ)
8. 抽出表との突き合わせ (1 項目ずつ検証して終わる)

不明点解消はタスクとして区切らない (随時、その場で聞くため)。

## WF の量的プロパティは表に固定する

コンポーネント名・タイポ階調・グリッド列数・縦横といった「だいたい合ってる」で通過しがちな値ほど後でずれる。WF が指定している値だけを会話内の表に固定する (推測値・読めない値は「不明」と書いてその場で聞く)。欠落要素 (ヘッダー無し等) も「無い」の 1 項目として残し、補完しない。この表が実装後の突き合わせの基準になる。

## データ仕様は別系統から取る (API 仕様 / DB スキーマ / 型定義)

どれが正なのかをユーザに聞く。型・必須性・文字数・列挙値・既定値の制約はクライアントバリデーションにも反映する。フロントだけ甘くするとサーバとの整合性が崩れて二重実装になるため。

## 既存実装の調査は必須。1 つに絞らず必要なものは全部読む

新パターンを発明すると一貫性が崩れるため、着手前に既存実装を読み切る。参考実装本体・配置先・兄弟コンポーネント・共通基盤・テストは全部対象になりうる。

- 「このプロジェクトで参考にすべき実装は？」をユーザに聞く
- 着手前に main を pull した上で、Explore agent を 2-3 並列で投げる
- 各 agent には「重要なファイルを 5-10 個リストアップ」と頼み、返ってきたものは省略せず全部読む
- 配置先の兄弟が参考実装と違うパターンなら、配置先に合わせる (兄弟と乖離するとレビューで戻され、後から書き直す手戻りが大きいため)

## 着手前に UI ライブラリの構造的制約を確認する

WF を読み込んだ時点で、実装方式と使用 UI ライブラリ (MUI / Mantine / shadcn 等) の内部構造を照合する。構造的に非互換なまま着手すると、スタイリング方法の試行錯誤が複数回発生するため。

非互換が判明したら、ハックのリスクと代替方針をセットでユーザに委ねる (どこまで WF に忠実にするか / どこから既存パターン優先かの判断はユーザの領分)。

## 不明点は生まれた瞬間に聞く

推測のまま実装に進むことが最大のリスク。読解中・調査中・データ仕様確認中、どのフェーズで生まれた不明点でもため込まず、その場でユーザに確認する (まとめて 1 度に聞く必要はない)。

ワイヤーフレーム固有でハマりやすい例: UI ラベル (「登録する」等の動詞) と API フィールド名の対応が読めない / disabled 要素が表示だけか送信もするか / 列挙値の選択肢順がコード列挙順と一致しない。

## 設計合意 → 実装委譲 → 抽出表と突き合わせて終わる

合意で出すもの: 変更ファイル一覧 / 設計判断のトレードオフ / 既存パターンとの対応 (「ここは X と同型」) / テスト観点。これを揃えずに実装委譲すると、TDD スキル側で前提を再質問することになり合意が崩れたまま着手することになる。テスト観点は参考実装より減らさない (見た目だけ寄せて検証密度が落ちる事故を防ぐため)。

合意後の実装は `superpowers:test-driven-development` に乗せる。バリデーションは純粋関数として export してテスト可能にする。

実装が戻ったら、読解時の抽出表を 1 項目ずつ突き合わせて初めて完了とする (「実装したら終わり」で抽出表が宙に浮くのを防ぐ)。確認手段は項目の性質で選ぶ — コードで確かめられる値 (列数・コンポーネント名) はマークアップ/スタイルを読んで照合する。見た目に出る値 (タイポ階調・余白・縦横) はレンダリングしてスクリーンショットを撮り WF と視覚比較する。ずれた項目は推測で埋めず WF / ユーザに差し戻す。

## コメントは WF を持たない読者向けに書く

コメントは「WF を持っていない読者」が読む前提で書く。コードを読む人がその WF / Figma / モックにアクセスできるとは限らないため。

外部成果物由来の値でも、コメントには「WF にこう書いてある」ではなく、値が成立する操作上・仕様上の根拠だけを書く。コメントの数値がコード中の値と重複していないかも確認する (重複していると、片方を変えたとき乖離が生じる)。
