# Implementation Review

変更されたコードを4つの観点で評価し、指摘事項を漏れなく報告する。

## ワークフロー

### Phase 1: コンテキスト収集

1. 変更対象ファイルを特定する（`git diff`, `git status`, または指定範囲）
2. 変更ファイルをすべて読む
3. 計画ファイルや仕様を探す:
   - ユーザが明示した場合はそのファイルを読む
   - 明示されていない場合は `~/.claude/plans/` 配下の最新ファイルや、会話中で言及された計画ファイルを探す
   - 見つからない場合はユーザに確認する。ただしサブエージェント起動時は確認をスキップし計画なしで進む（対話不可のため）
4. **サブエージェント（Explore）を使い、既存コードベースから以下を並列収集する:**
   - 変更ファイルと同ディレクトリ・同ドメインの類似コンポーネント
   - 変更で使われているパターンの他箇所での使用例
   - プロジェクト固有の慣習（命名、ファイル構成、テストパターン等）

### Phase 2: 4観点レビュー

**重要度が低い指摘も省略しない。** typo、微妙な命名、軽微な不整合もすべて報告する。

#### 観点1: モダンさ — 古い文法・非推奨パターンの検出

- 言語・フレームワークの現行ベストプラクティスと照らして古い書き方がないか
- より簡潔・安全・パフォーマントな代替構文がないか
- ただし **プロジェクト全体が古いパターンで統一されている場合は、そのパターンに従うのが正解**
- 判断基準: 「ここだけ変えて改善になるか」

#### 観点2: 意図の明確さ — 読み手に伝わる書き方か

- 命名が動作・目的を適切に表しているか
- 処理の流れが追いやすいか（早期リターン、ガード節、適切な分割）
- コメントの過不足
- マジックナンバーやハードコード値

#### 観点3: 既存コードとの一貫性

**最重要観点。** Phase 1 で収集した類似コードと差分を並べて比較し、同じ状況で異なるパターンを使っている箇所をすべて洗い出す。

チェック項目: エラーハンドリング、状態管理、UI構成、テスト記法、API層のパターン。

**リファクタリング・共通化時の追加チェック:**

- **利用コンテキストとの整合:** 抽出した関数・クラス・コンポーネントが、すべての呼び出し箇所のコンテキストで正しく動作するか。呼び出し元が前提とする制約（型、レイアウト、スレッド安全性、可視性など）と矛盾しないか確認する
- **削除・リネーム後の参照残り:** 削除・移動した識別子（変数、定数、クラス名、設定キーなど）をテストや他ファイルが参照したまま残っていないか。参照先が未定義になりテストが偽陽性になるケースに注意
- **フォールバック戦略の統一:** 同じ値に対するフォールバックやデフォルト値の扱いが全箇所で統一されているか。表示用と内部用で挙動が乖離していないか
- **既存パターン踏襲時の妥当性:** 兄弟コンポーネントのパターンをコピーするとき、そのパターンが過去 PR で「部分対応」にとどまっていないか確認する。表示ガード（例: `!error` で描画を抑制）は state 残存・タイミング競合・未解放リソースなどの根本問題を隠蔽しがちで、パターンをそのまま複製しても問題は引き継がれる。**慣習に従うこと自体は正解だが、慣習の由来 PR を読み、そのパターンが本来の要件を完全に満たしているかを必ず問う**
- **コピーした固定値・キー名の整合確認:** 他ファイルからコピーした固定値やキー名は、コピー元との値のリテラル一致と、そのキーを読む実装側コードの存在を確認する（未参照フィールドはテストの pass/fail では検出できない）

#### 観点4: 非機能要件 — パフォーマンス・堅牢性の過不足

**不足:** 不要な再計算・再レンダリング、リソースリーク、レースコンディション、N+1的な非効率

**過剰:** 不要なメモ化、過度な抽象化、不要な防御的コピー、起こりえないケースへのエラーハンドリング

### Phase 3: レポート出力

```
## コードレビュー結果

### 1. モダンさ
[指摘]

### 2. 意図の明確さ
[指摘]

### 3. 既存コードとの一貫性
[指摘]

### 4. 非機能要件
[指摘]
```

各指摘には以下を含める:

| 項目 | 内容 |
|------|------|
| 箇所 | `ファイルパス:行番号` |
| 内容 | 何が問題で、なぜ問題か |
| 提案 | 具体的な修正コードまたは方針 |
| 重要度 | 高 / 中 / 低（`.claude/skills/shared/review-severity.md` の定義に従う） |

指摘がない観点は「指摘なし」と一言書いて次に進む。

## 原則

- **根拠なき指摘をしない。** 「このプロジェクトではこうしている（具体例: ファイルパス:行番号）から揃えるべき」と示す
- **慣習 vs ベストプラクティスが異なる場合:** 明らかにどちらかが優れていれば根拠付きで提案。微妙な場合は両方のメリット・デメリットを併記し判断を委ねる
- **重要度は `.claude/skills/shared/review-severity.md` の定義に従う。** 実害がない指摘を中以上にしない
