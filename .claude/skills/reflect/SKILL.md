# タスクふりかえり — retro + session-feedback を 1 パスで統合起動

**main は生ログを読まない**（§2 で隔離サブエージェントに委譲）。送信先が PUBLIC であることの漏洩規律は retro 冒頭と同じ（参照）。

`--auto` 付き起動（無人モード）は §1〜§5 に §6 の差分を重ねて実行する。`--auto` の最終メッセージは 0 件・エラー時も例外なく §6 のマーカー形式だけで返す（読み手は人間でなくパーサ）。引数なしの対話版に §6 は適用しない。

## §1. 事前チェック

`retro` SKILL.md §1 をそのまま実行する（参照。写経しない）。

## §2. 抽出（retro-extractor を 1 回起動）

`retro-extractor` エージェントを 1 回起動する。渡すものは retro §2 の一式に加え、次の 3 つを追加する:

1. **このファイル（reflect SKILL.md）の絶対パス** — extractor に本節の B 仕様・A/B 振り分けルールを追加の作業仕様として読ませる
2. **機械抽出コマンドの実行指示**: `session-feedback-extract <§1 で特定した jsonl の絶対パス>` を実行させる（読み取り専用スクリプト。出力内容の正は session-feedback §1.1）。出力を B 系シグナルの一次材料として使わせる
3. **session-feedback SKILL.md の絶対パス** — B の Category 判定に §1.2 のカテゴリ表を読ませる

### 返却フォーマット

```text
## A: スキル指摘（サニタイズ済み・公開可）
### 指摘 1
（retro §2.2 のスキーマ）
...
指摘件数: <N>

## B: 行動・記憶（ローカル適用のみ・非公開）
### 行動 1
**分類/推奨先:** <§1.2 のカテゴリ> / <保存先の対応表の場所 1 つ（新規 or 更新）>
**内容:** <起きたこと + 根本原因を 1 文>
**書き込み案:** <書く本文そのもの。最小限。既存行の変更は before/after のみ>
...
行動件数: <M>

B は簡潔に書く。
```

### A/B の振り分けルール

dotfiles 管理スキルの SKILL.md の記述（または欠如）に帰属できるシグナルは **A のみ**（process-retro が直すので B に重複させない）。帰属できない行動・記憶・ドキュメント不整合・権限見直しは **B**。session-feedback の「使用スキル分析」のうち dotfiles スキルの本文改善は A に吸収され、新規スキル候補・トリガー漏れだけが B（案内のみ）に残る。

### B の採用バー

採用基準は session-feedback §1.2「抽出のコツ」に従う（参照）。reflect 固有の追加: 一回性（その場の文脈・好みに留まるもの）は落とす。推奨アクションは 1 件につき 1 つに絞る（複数案の併記はしない — 往復を消すのが目的）。

B は原則サニタイズ不要（ローカルにしか書かない）。**例外**: 書き込み先が dotfiles 管理ファイル（`~/.claude/CLAUDE.md` 等、PUBLIC リポに追跡されるもの）の B 項目は、書き込み案にも retro §3 のサニタイズを適用する。A に入るのは A ブロックだけという分離は絶対（issue 本文組み立てで B を混ぜない）。

エラー返却形は retro §2 と同じ（参照）。retro §2.3 の 0 件スキップは **A・B とも 0 件のときだけ**適用する（片方だけ 0 件でも §3 の確認は行う）。

## §3. 一括確認（1 往復）

extractor の返却を**一字一句そのまま**テキスト出力する（main は要約・整形しない。A はこのプレビューが唯一のサニタイズ検知機会（retro §4 と同じ）、B はユーザが書き込み原文を確認できる唯一の機会）。続けて 1 つのプロンプトで両方の判断を仰ぐ（`AskUserQuestion` は使わない）:

```text
送信先: crgstar/dotfiles（GitHub issue・PUBLIC）
選んだ jsonl: <§1 で特定したパス>
A: このまま送ってよいですか？（送る / 修正指示 / やめる）
B: 適用する番号を指定してください（例: 全部 / 1,3 / なし）
```

修正指示が来たら反映して**全ブロックを再掲**して仰ぎ直す（retro §4 と同じ）。ただし修正指示でも **B の内容を A に移さない**（main はサニタイズできない。A に載せたいシグナルは §2 の抽出からやり直す）。

## §4. 適用

**A**: retro §4「本文の組み立てと送信」に従う（参照）。issue 本文に入れるのは A ブロック内の `### 指摘 N` 群と `指摘件数:` 行だけ（`## A:` 見出しは外す。process-retro のパーサ互換のため）。

**B**: 承認された番号だけ書き込む。memory は auto-memory 規約（MEMORY.md index 更新含む）、CLAUDE.md / 既存ファイル編集は session-feedback §3.4 の注意に従う（参照）。

`~/.claude/CLAUDE.md` は dotfiles（PUBLIC）管理なので、追記内容に作業リポ固有情報を書かない（コミット時の sanitize 規約が最終防壁だが、書かないのが先）。

末尾サマリ（必ず 1 行）:

```text
ふりかえり: スキル指摘 <N> 件（送信済み|送信なし|失敗） / 行動 <M> 件中 <K> 件適用
```

## §5. エラー処理

A 系は retro §5 に準拠（参照）。B 系は項目単位で失敗を記録してスキップし、実行全体は止めない。このスキルのエラーで他の作業をブロックしない。

## §6. 無人モード（`--auto <transcript>`）

`/reflect --auto <セッション jsonl の絶対パス> [--since-line <N>]` は `run-headless.sh` からのヘッドレス起動専用。対話版との差分だけ示す（示していない箇所はすべて対話版と同じ）。

- **§1 の置換**: jsonl は引数の transcript を使う（`CLAUDE_CODE_SESSION_ID` は別セッションなので使わない）。retro §1.1 の `gh auth status` はスキップ（投稿はドライバの責務）。retro §1.4 の mirugit annotation はスキップ（終了済みセッションの token は特定できない）。
- **`--since-line <N>` 付きの再処理**: このセッションは行 N まで処理済み。extractor へ渡す指示に次を追加する: 「jsonl の行 1〜N は前回処理済み。文脈として読んでよいが、**行 N 以前のやりとりだけに根拠を持つシグナルは A・B とも拾わない**。行 N より後に根拠（訂正・反復・拒否等の発話）が現れたものだけを対象にする」。境界を跨いで同根のシグナルが再抽出された場合の保険は retro §1.5 の既存 open issue 重複チェック（従来通り実施）。
- **§3 の置換（人間プレビューなし）**: extractor の返却をテキスト出力せず、A ブロック全文を `sanitize-auditor` に **2 回、独立に**検査させる（`Agent` ツールで別々に起動し、一方の結果を他方に渡さない）。渡す入力は (1) 検査対象 = A ブロック全文（diff の代わり） (2) 判定基準 = `~/dotfiles/.claude/skills/process-retro/SKILL.md` の絶対パス。判定:
  - **両方 clean** → A は「投稿可」（REFLECT-OUTBOX で返す）
  - **どちらか一方でも漏洩あり・判断不能・返却不正** → A 全体を「保留」（REFLECT-HOLD で返す。部分投稿はしない）
- **§4 の置換（書き込み・投稿をしない）**: Write / `gh api` / `rm` を使わない（権限もない）。結果は下記マーカーの最終メッセージだけで返し、ファイル保存・issue 投稿・再送はドライバが行う。
- **B は適用しない**（無人化は A のみ。B ブロックは REFLECT-SUMMARY に載せ、人間が朝サマリを見て対話版で適用する）。
- **出力形式の全面上書き**: 対話版が規定する素のテキスト出力 — §2.3 の 0 件スキップ時の末尾サマリ、§4 の末尾サマリ、§5 / retro §5 のエラー報告 — は**すべて下記マーカー形式に置き換える**。0 件でも REFLECT-SUMMARY マーカーで返す（素の 1 行サマリを返すとドライバがパース不能として hold に落とし、0 件のたびに人間確認が発生してしまう）。

### 返却フォーマット

最終メッセージはマーカーブロックだけで構成する（マーカー外に文章・前置き・説明を一切置かない。各マーカーは単独行）。

投稿可の A があるとき（1 行目は `title:`、2 行目以降が issue 本文。本文の形は retro §4「本文の組み立てと送信」と同じ）:

```text
<<<REFLECT-OUTBOX
title: [ふりかえり] dotfiles スキル: <N> 件の指摘（<YYYY-MM-DD>）
# dotfiles スキル ふりかえり（自動生成）
<§2.2 の指摘ブロック群 + 指摘件数行>
REFLECT-OUTBOX>>>
```

保留のとき（A ブロック全文 + 監査 2 回の判定要旨。人間が朝に判断する材料）:

```text
<<<REFLECT-HOLD
<A ブロック全文>
<監査1・監査2 の判定要旨>
REFLECT-HOLD>>>
```

常に最後に（A・B とも 0 件でもこれだけは返す）:

```text
<<<REFLECT-SUMMARY
ふりかえり: スキル指摘 <N> 件（投稿可 | 保留 | 0 件） / 行動 <M> 件（未適用・対話版で処理）
<B ブロック全文（M ≥ 1 のとき。ローカルの朝サマリにしか書かれない）>
REFLECT-SUMMARY>>>
```

エラー時（§5 相当）も `ふりかえり: エラー — <一行の説明>` を REFLECT-SUMMARY で返す。SUMMARY を欠いた出力はドライバがパース不能として保留に落とす。
