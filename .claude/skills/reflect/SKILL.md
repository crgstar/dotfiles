# タスクふりかえり — retro + session-feedback を 1 パスで統合起動

**main は生ログを読まない**（§2 で隔離サブエージェントに委譲）。送信先が PUBLIC であることの漏洩規律は retro 冒頭と同じ（参照）。

## §1. 事前チェック

`retro` SKILL.md §1 をそのまま実行する（参照。写経しない）。

## §2. 抽出（retro-extractor を 1 回起動）

`retro-extractor` エージェントを 1 回起動する。渡すものは retro §2 の一式に加え、次の 3 つを追加する:

1. **このファイル（reflect SKILL.md）の絶対パス** — extractor に本節の B 仕様・A/B 振り分けルールを追加の作業仕様として読ませる
2. **機械抽出コマンドの実行指示**: `session-feedback-extract <§1 で特定した jsonl の絶対パス>` を実行させる（読み取り専用スクリプト。出力内容の正は session-feedback §1.1）。出力を B 系シグナルの一次材料として使わせる
3. **session-feedback SKILL.md の絶対パス** — B の Category 判定に §1.2 のカテゴリ表を読ませる

### 返却フォーマット

```text
## A: スキル指摘（サニタイズ済み・公開可）
### 指摘 1
（retro §2.2 のスキーマ）
...
指摘件数: <N>

## B: 行動・記憶（ローカル適用のみ・非公開）
### 行動 1
**分類/推奨先:** <§1.2 のカテゴリ> / <保存先の対応表の場所 1 つ（新規 or 更新）>
**内容:** <起きたこと + 根本原因を 1 文>
**書き込み案:** <書く本文そのもの。最小限。既存行の変更は before/after のみ>
...
行動件数: <M>

B は簡潔に書く。
```

### A/B の振り分けルール

dotfiles 管理スキルの SKILL.md の記述（または欠如）に帰属できるシグナルは **A のみ**（process-retro が直すので B に重複させない）。帰属できない行動・記憶・ドキュメント不整合・権限見直しは **B**。session-feedback の「使用スキル分析」のうち dotfiles スキルの本文改善は A に吸収され、新規スキル候補・トリガー漏れだけが B（案内のみ）に残る。

### B の採用バー

採用基準は session-feedback §1.2「抽出のコツ」に従う（参照）。reflect 固有の追加: 一回性（その場の文脈・好みに留まるもの）は落とす。推奨アクションは 1 件につき 1 つに絞る（複数案の併記はしない — 往復を消すのが目的）。

B は原則サニタイズ不要（ローカルにしか書かない）。**例外**: 書き込み先が dotfiles 管理ファイル（`~/.claude/CLAUDE.md` 等、PUBLIC リポに追跡されるもの）の B 項目は、書き込み案にも retro §3 のサニタイズを適用する。A に入るのは A ブロックだけという分離は絶対（issue 本文組み立てで B を混ぜない）。

エラー返却形は retro §2 と同じ（参照）。retro §2.3 の 0 件スキップは **A・B とも 0 件のときだけ**適用する（片方だけ 0 件でも §3 の確認は行う）。

## §3. 一括確認（1 往復）

extractor の返却を**一字一句そのまま**テキスト出力する（main は要約・整形しない。A はこのプレビューが唯一のサニタイズ検知機会（retro §4 と同じ）、B はユーザが書き込み原文を確認できる唯一の機会）。続けて 1 つのプロンプトで両方の判断を仰ぐ（`AskUserQuestion` は使わない）:

```text
送信先: crgstar/dotfiles（GitHub issue・PUBLIC）
選んだ jsonl: <§1 で特定したパス>
A: このまま送ってよいですか？（送る / 修正指示 / やめる）
B: 適用する番号を指定してください（例: 全部 / 1,3 / なし）
```

修正指示が来たら反映して**全ブロックを再掲**して仰ぎ直す（retro §4 と同じ）。ただし修正指示でも **B の内容を A に移さない**（main はサニタイズできない。A に載せたいシグナルは §2 の抽出からやり直す）。

## §4. 適用

**A**: retro §4「本文の組み立てと送信」に従う（参照）。issue 本文に入れるのは A ブロック内の `### 指摘 N` 群と `指摘件数:` 行だけ（`## A:` 見出しは外す。process-retro のパーサ互換のため）。

**B**: 承認された番号だけ書き込む。memory は auto-memory 規約（MEMORY.md index 更新含む）、CLAUDE.md / 既存ファイル編集は session-feedback §3.4 の注意に従う（参照）。

`~/.claude/CLAUDE.md` は dotfiles（PUBLIC）管理なので、追記内容に作業リポ固有情報を書かない（コミット時の sanitize 規約が最終防壁だが、書かないのが先）。

末尾サマリ（必ず 1 行）:

```text
ふりかえり: スキル指摘 <N> 件（送信済み|送信なし|失敗） / 行動 <M> 件中 <K> 件適用
```

## §5. エラー処理

A 系は retro §5 に準拠（参照）。B 系は項目単位で失敗を記録してスキップし、実行全体は止めない。このスキルのエラーで他の作業をブロックしない。
