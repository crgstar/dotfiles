# Sentinel: 多角コードレビュー

## フェーズ 1: 差分の特定

差分は `git diff`（ステージ済みなら `git diff HEAD`）→ 空ならこの会話で編集したファイル → それも無ければユーザーに確認する。**ブランチ差分まで勝手に拡げない**（スコープ越境を防ぐ）。

差分中のテストファイル（`*.test.*` / `*.spec.*`）の有無で、フェーズ 2 のテスト設計エージェントを起動するか判定する。同様に、差分中にコメント行（`//` / `/* */` / `{/* */}`）の追加・変更があればコメント精査エージェントを起動する。

## フェーズ 2: レビューエージェントを並列起動

**1 メッセージにまとめて同時起動**し、各エージェントに完全な差分を渡す。

| # | エージェント | 起動条件 | subagent_type | 入口スキル |
|---|---|---|---|---|
| 1 | 実装品質 | 常時 | general-purpose | `/implementation-review` |
| 2 | テスト設計 | テスト差分あり | general-purpose | `/test-design-guide` |
| 3 | コメント精査 | コメント差分あり | general-purpose | `/comment-scrutiny` |
| 4 | バグ・規約レビュー | 常時 | `feature-dev:code-reviewer` | （agent 自身の system prompt） |

**type 使い分けの why**: 1〜3 は Skill を呼ぶため Skill ツールが使える general-purpose が必須。4 は named agent（tools 制限・model 固定済み）で Skill 呼び出し不要。

各エージェントへの追加指示:

- **1. 実装品質**: プロンプトに以下を含める。
  - 既存コードとの一貫性を最優先（指摘には根拠ファイル:行を併記）
  - 非機能要件は「不足」だけでなく「過剰」（不要なメモ化・過度な抽象化）も検出
  - 重要度は `.claude/skills/shared/review-severity.md` の定義に従う。**本文をプロンプトに直接含める**（独立コンテキストの subagent には参照パスが届かないため）
  - コメント追加の提案は `/comment-scrutiny` 通過後に絞る（提案→全削除の二重審査を防ぐ）
- **2. テスト設計**: `/test-design-guide` に従う（観点はそちらに集約）
- **3. コメント精査**: `/comment-scrutiny` に従う。差分内のコメントを「消すと壊れる」かで分類
- **4. バグ・規約レビュー**: `feature-dev:code-reviewer`（named agent・tools 制限済み・Bash なし）に差分全文をプロンプトで渡す。agent は CLAUDE.md 準拠・バグ・セキュリティを confidence≥80 で絞って報告する。出力は Critical / Important の 2 段で日本語

## フェーズ 3: 一次情報との突き合わせ（必須）

エージェントは独立コンテキストで動くため、(a) **設計書 / 仕様書 / 参考文献まとめ**（docbase 等の外部設計書、`.local/*-references.md`、baton 内で参照された一次情報）と (b) **兄弟ファイル**（同ディレクトリの `*Dialog.tsx` / `*.test.tsx` / 類似実装）との突き合わせが弱い。集約前に両方を Read / Grep で確認し、各指摘を「**仕様書で確定済＝即スキップ or 即対応 / 既存と一致＝慣習の追認 / 既存と不一致＝改善余地**」に分類する。**省略すると判断をユーザに丸投げした「要検討」を量産することになる。**

差分に UI カラーリテラルがあれば、類似コンポーネント・関連画面を横断検索して既存の色値やデザイントークンとの整合を確認してからフェーズ4に進む。why: カラーリテラルはプロジェクト全体でのデザイントークン管理が多く、局所差分だけでは不整合を見逃しやすいため。

判定不能（一致と不一致が混在 / 該当パターンが新規でルール不明）のときだけ `gh pr list --search "<keyword>"` / `git log -p <path>` で関連 PR を探し、議論経緯を HTML の「過去 PR の議論」セクションに添える。**自動では走らない**（無関係 PR を漁って指摘の根拠を捏造するリスクがあるため）。

**既存候補が 2 種類以上ある場合**は、変更日時の新しい方を「現行慣習」として採用する（差が微妙なら採用数が多い方）。さらにその実装の PR を `gh pr view` で確認し、議論があればその結論を推奨に反映する。why: 「既存」の定義がブレると下の推奨表の判定も揺れるため、まず「何を既存とみなすか」を決める。

### 推奨の計算（既存重視）

| 既存との関係 | 重要度 | 推奨 | フェーズ 4 で期待する選択 |
|---|---|---|---|
| 一致（兄弟全体が同じ状態） | 中 / 低 | **スキップ推奨**（慣習追認） | `skip` |
| 一致（兄弟全体が同じ状態） | 高（セキュリティ / データ損失等のブロッキング） | **対応推奨**（全体一致 → 合わせて検討） | `fix` |
| 不一致 | 高 / 中 / 低 | **改善推奨** | `fix` |
| 不一致（影響が読めない） | 任意 | **要検討** | `defer` |
| 比較対象なし（新規パターン） | 任意 | **要検討**（過去 PR 議論を併記） | `defer` |

## フェーズ 3.5: 先行修正（auq-web に載せない）

振る舞いを変えない指摘は先行修正する。why: 確認コストに見合わない（やらないより必ずやった方がいい類）。

対象: **テスト追加・テストヘルパー追加 / コメントのみの修正**。既存テストの期待値を変える修正は対象外（仕様判断を含み得るため auq-web へ回す）。迷ったらフェーズ 4 へ。

先行修正した内容はチャット（通常の出力）で報告する。auq-web の HTML には載せない（既に終わっている事実報告であり、対応可否を問う対象ではないため）。

## フェーズ 4: auq-web で「説明 + 対応可否」を 1 画面に

`auq-web` スキルに従い、**指摘 1 件 = 質問 1 件**として並べる（集約すると排他選択と根拠の紐付けが弱まるため指摘単位で独立に取る）。各質問の desc にその指摘の「既存 vs 今回」コード並列・推奨・why を書く。チャットには長い MD は出さない。

### sentinel 固有の点（実行は `auq-web` の手順に従う）

- **meta は省略**: 冒頭ヘッダー（対象差分・各エージェント一行サマリ）を **「最初の指摘の desc 冒頭」** にまとめる
- **指摘 N 件 → question N 個** を 1 つの HTML に並べる
- `event: "reject"` のときはフェーズ 5 へ進まない

### 各 question の構成

- `id: "issue_<n>"` / `kind: "single"`（multi にすると「対応 / スキップ」が排他で取れないため single 固定）
- `title`: `#<n> [<agent略称>/重要度] 一行サマリ（出典）`
  - agent 略称はフェーズ 2 の番号順に `実装` / `テスト` / `コメント` / `バグ規約`
- `options[]`:
  - `{ value: "fix",     label: "対応する" }`
  - `{ value: "skip",    label: "スキップする" }`
  - `{ value: "defer",   label: "保留して議論したい" }`
  - `{ value: "explain", label: "説明が足りないのでもっと説明して" }`
- `explain` 選択時の不足点は共通 comment 欄に書いてもらう運用

### 各 question の desc HTML

- **最初の指摘の desc 冒頭のみ**: 対象差分のファイル一覧 / 各エージェントの一行サマリ
- 既存との比較: 一致 / 不一致 + 具体ファイル:行
- 「既存 vs 今回」コード並列: 兄弟ファイル該当箇所と今回の差分該当箇所を `<pre><code>` 2 つで並べ、判断を「記述より現物」に寄せる
- 推奨 + why（既存追認 / 改善推奨 / 高重要度のため対応推奨 など）
- 過去 PR の議論セクション（該当時のみ）: PR 番号・論点・結論未収束ならその旨

## フェーズ 5: 説明ループ → 修正

回答を受けたら **まず `explain` がある指摘を処理する**:

- `selected: "explain"` の指摘について、追加で Read / Grep / 兄弟検索 / `gh pr list --search` 等で根拠を集め、当該 question の desc を厚くしてフェーズ 4 を再起動する（comment に書かれた不足点に答える形で desc を更新）
- 他の指摘について既に答えられた選択は維持する（ユーザーが答えた選択を尊重するため、当該 question 自体を HTML から除外して再提示してもよい）
- ループ周回数を会話内でカウントし、各再提示時に「説明ループ N 周目」と明示する。停止条件:
  - すべての回答が `fix` / `skip` / `defer` になった → 修正フェーズへ
  - ユーザーが reject した → フェーズ 5 を抜けてユーザーに意図確認
  - 3 周到達 → 修正フェーズには進まず、残った `explain` 指摘と最新の comment をチャットに 1 ブロックで提示し口頭で詰めてもらう（説明し続けても合意できない場合は Claude 側の素材不足なので一旦止める）

すべての `explain` が解消されたら修正フェーズへ進む。**`fix` のみコードを修正し、`defer` は保留として控え、`skip` は無視**。各 `comment` の補足があれば反映する。修正完了後、対応 / スキップ / 保留の振り分けをチャットに 1 ブロックで報告。
