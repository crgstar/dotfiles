# Session Feedback — 反省・改善ふりかえりスキル

セッション中に Claude が犯した失敗や、もっと効率よくできたはずのやりとりを洗い出し、**保存先まで選び切ってから**永続化レイヤに記録する。

## なぜ必要か

同じ失敗を次回も繰り返すのは、気付いても言語化せず・保存しないから。会話の流れで気付いた断片を、セッション終盤で構造化して memory や CLAUDE.md に還元することで、再発を防ぐ。

特に拾いたい 2 パターン:

- **繰り返した失敗** — 同じセッション内で同種の間違いを複数回やった。一度きりのミスより優先度が高い
- **不要な確認** — ユーザーに聞いたが、実際はコード・ドキュメント・既存の memory を調べれば自力で解決できた。ユーザーの時間を無駄にしている

このスキルの主眼は **Claude 自身の反省点**の言語化。ユーザーへの要望やセッションの良かった点は扱わない。

## ワークフロー

### 1. 抽出

#### 1.1 機械抽出 (必ず最初にやる)

会話の主観だけだと見落としや圧縮による欠損があるため、付属スクリプトで先に事実を集める。

```bash
session-feedback-extract
```

得られるもの:
- session span (ユーザー turn 数 / assistant turn 数 / 期間)
- 発動した Skill 一覧 (`Skill` ツール経由のみ。SessionStart で auto-load される `using-superpowers` 等は含まれない)
- ユーザーが拒否した tool_use 一覧 ("doesn't want to proceed" シグナルで検出。ask→allow の記録は jsonl に残らないので拾えない)

このあとの主観抽出 (1.2) は、この出力を必ず参照しながら進める。

#### 1.2 主観抽出

会話履歴を読み、以下のカテゴリで具体的な改善点を集める。**無ければ無理に埋めない**。

- **繰り返した失敗** — 同じ性質のミスを 2 回以上やった瞬間。どこで気付くべきだったか、根本原因は何かを添える
- **不要な確認** — ユーザーへの質問のうち、手元の情報で解決できたもの。「何を見れば分かったか」を書く
- **その他の改善点** — ユーザーから修正が入った行動、遠回りな手順、非効率な判断
- **使用スキル分析** — 1.1 で得た発動スキル一覧を起点に、(a) 各スキルが今回の文脈で適切に機能したか、(b) トリガー条件・本文の改善案、(c) 同種の作業を繰り返したのにスキル化されていない領域 (新規スキル候補) を洗い出す
- **権限拒否ログ** — 1.1 の denied 一覧をそのまま提示する。**Claude は「許可すべきだった」「拒否で正しい」の判定をしない**。ユーザーが見て settings に反映するかを判断する
- **ドキュメント更新候補** — 今セッションで触ったコード/挙動と、リポジトリ内のドキュメント (CLAUDE.md 以外: `docs/` 配下、`README.md`、各 `SKILL.md`、その他 `.md`) との整合を確認し、追記・修正すべき箇所を挙げる

抽出のコツ:

- 各項目について「**これを保存しなければ、次回以降 Claude は再び同じミスをするか？**」を問う。No なら記録候補から外す。No になりやすいパターン:
  - 空疎な一般論 (「コードは綺麗に書いた方がいい」のような、このセッション中の具体的な瞬間にひも付かないもの)
  - 根本原因まで言語化できず、保存しても再発防止できないもの
  - 一度きりの偶発ミス (繰り返し性がない)

- 失敗の**症状だけでなく根本原因**も一緒に書く (なぜ気付けなかったか、何を参照すべきだったか)

- 保存候補として残った各項目は、**既存の永続化先すべてと照合する**: `~/.claude/CLAUDE.md` / `<project>/CLAUDE.md` / `<project>/CLAUDE.local.md` / `.claude/rules/*.md` / auto memory。同趣旨のルールがヒットした項目は「既存ルールあり ("...")、守れなかった」と明記し、**再発を防ぐための改善案を併せて提示する**（書き方は §3）。改善案が思いつかなければ改善案なしで報告するに留め、次回の自律的遵守に任せる

### 2. 提示

以下の例のように一覧を出す。各項目は 2-4 行程度で、該当箇所の短い引用と根本原因を添える。**可読性向上のためカテゴリごとに表形式での提示を推奨**。各セルは 1 行で収め、書ききれない詳細は表外に箇条書きで補足する (Claude Code の表レンダラーは `<br>` や行末 2 スペース等のセル内改行をサポートしない: GitHub issue #26390)。

```
## 繰り返した失敗
1. [内容] — 根本原因: [why] / 推薦先: [場所]

## 不要な確認
1. [内容] — 本来の解決手段: [どこを見れば分かったか] / 推薦先: [場所]

## その他の改善点
1. [内容] — 推薦先: [場所]

## 使用スキル分析
発動したスキル: foo (3回), bar (1回)
- foo — 評価: 適切に機能 / 改善案: なし
- bar — 評価: トリガー漏れ (本来 X の文脈で発動すべきだったが起動せず) / 改善案: description に「Y のとき」を追記
- (新規スキル候補) Z 系の手作業を 4 回繰り返した。スキル化する価値あり

## 権限拒否ログ (判定なし・参考情報)
- {"name":"Bash","input":{"command":"gh api ...","description":"..."}}
- {"name":"Edit","input":{"file_path":"..."}}

## ドキュメント更新候補
- docs/setup.md — `setup.sh` の env 引数説明が今回の変更後の挙動と一致しない
- README.md — 新規追加した extract.sh の存在に触れていない
```

提示後、「追加・修正はありますか？」と一度だけ聞き、ユーザーの入力を反映してリストを確定する。

### 3. 保存先の推薦

保存先は **2 軸**で決まる。各項目で両軸 + 理由を必ず明示する。

**軸 1: スコープ** — ユーザー全体 / プロジェクトのみ
- 別リポジトリでも効かせたい → ユーザー全体
- このリポジトリ内限定 → プロジェクトのみ

**軸 2: 強度** — 常時 / 関連時
- 毎セッション確実に参照させたい → 常時 (CLAUDE.md)
- 関連作業のときだけで十分 → 関連時 (memory)

|              | ユーザー全体               | プロジェクトのみ                                        |
|--------------|--------------------------|--------------------------------------------------------|
| 常時 (強)     | `~/.claude/CLAUDE.md`    | `<project>/CLAUDE.md` (共有) / `CLAUDE.local.md` (ローカル) |
| 関連時 (弱)   | (該当なし)               | auto memory                                             |

memory は project scope のみ。type は内容で選ぶ: **feedback** (行動ルール)、**user** (属性)、**project** (作業状況)、**reference** (外部ポインタ)。

**提示の形式** — 各項目で以下を書く:

> [反省内容] — 推薦: `<場所>` / スコープ: <ユーザー全体 | プロジェクトのみ> / 強度: <常時 | 関連時> / 理由: <why>

**既存ルールに該当する項目の扱い**

§1 で「既存ルールあり、守れなかった」とタグ付けされた項目は、**新規保存ではなく既存ルールへの改善案**を提示する。

まず **なぜ既存ルールが今回発動しなかったか** を分析する (トリガー条件が主観的自覚に依存していた / 射程が暗黙に限定的だった / 実例不足で抽象すぎた、等)。原因を特定できれば改善案はほぼ見える。原因が本当に特定できないときに限り「改善案なし」に倒す。

改善案の中心は **ルール文そのものの抽象度の再設計** である。

抽象度が高すぎると平凡な原則になって発動しない。狭すぎると今回のケースしかカバーできず、近接する失敗で再発する。「**過去にあった同種の失敗・未来に起こりうる同種の失敗を、過不足なく覆う**」抽象度を狙う。一般性を維持したまま射程を明確化できる文を先に探し、射程を狭める方向に倒すときは適切な理由を要する。

抽象度の再設計で十分な改善案が書けないとき、**最後の手段**として実例を本文に併記する。実例は具体例として強力だが時間とともに陳腐化し、ルールの核を見失わせやすい。実例に頼る前に、抽象度の言い直しで同等の効果が出ないかをまず考える。

既存ルールが重複・矛盾しているなら、削除・統合も改善案として検討する。

改善案は **複数案を並べて比較検討** する。各案の効果と懸念を短く添え、単一案で済ませない。採用可否はユーザーに判断してもらう。

**その他**
- まとまった手順・ガイド → 新規スキル or Markdown
- 自動検知可能な振る舞い → 保存は任意、末尾で hookify 候補として案内

**新カテゴリ専用の保存先**

- **使用スキル分析**
  - 軽微な追記・本文の補強 → 該当 `~/dotfiles/.claude/skills/<skill>/SKILL.md` を直接編集
  - トリガー漏れ / description の精度改善 → 直接編集せず `/skill-creator` を案内する。skill-creator は description 用の eval ループ (`run_loop.py`) を持っているので、勘で書き換えるより検証付きで最適化したほうが確度が高い
  - 新規スキル候補 → このスキル内では作らない。「`/skill-creator` を別途起動してください」と案内するだけ
- **権限拒否ログ** → 保存しない。リスト出力で終わり。settings 反映の判断はユーザーが行う
- **ドキュメント更新候補** → 該当ファイル (例: `docs/setup.md`, `README.md`, 該当 `SKILL.md`) を直接編集

**迷ったら**
- スコープが迷ったら **ユーザーに確認する**
- 強度が迷ったら **memory** を優先 (CLAUDE.md を肥大化させない)

### 4. 一件ずつ保存

リストの先頭から順に処理する。各件で:

1. 推薦先を再掲し、「ここに保存しますか？ (Y / 別の場所 / スキップ)」と聞く
2. 変更要求なら別の保存先を提示
3. 承認されたら書き込み、進捗を「3/8 保存済み」のように示す

**一括保存はしない**。一件ずつ判断を挟むことで、ゴミが memory/CLAUDE.md に混入するのを防ぐ。

書き込み時の注意:

- memory は `auto memory` システム (グローバル CLAUDE.md 参照) の規約に従う。feedback memory は `**Why:**` `**How to apply:**` を付け、`MEMORY.md` の index 更新も忘れない
- CLAUDE.md は既存の章立てを尊重し、関連セクションに追記する。新しいトップレベル見出しを乱造しない
- 相対日付 (「今週」「Thursday」) は絶対日付に変換してから保存する
- **既存エントリとの重複チェックは書き込み前に必ず行う**。同趣旨のルールがすでにあるなら更新 (追補・具体化) に倒す

カテゴリ別の処理:

- **使用スキル分析 (軽微な改善)** → 該当 `SKILL.md` に直接 Edit。本文への補足追記・誤字修正などはセクション単位で。承認時に該当箇所を提示してから書き込む
- **使用スキル分析 (トリガー漏れ / description 精度改善)** → 直接 Edit せず `/skill-creator` を案内する。description はトリガー精度に直結するため、eval で検証せずに勘で書き換えると逆にトリガーしなくなるリスクがある
- **使用スキル分析 (新規スキル候補)** → 「`/skill-creator` を別途起動してください」と案内するのみ。**このスキルは新規 SKILL.md を作らない**
- **権限拒否ログ** → 保存処理にかけない。§2 のリストを出した時点で完了。「settings 反映するかは判断してください」と添える
- **ドキュメント更新候補** → 該当ファイルを Read してから関連箇所に Edit。新しいトップレベル見出しは乱造しない

### 5. hookify 候補の紹介

リスト処理が終わったら、**自動検知で強制できると価値があるもの**を列挙する:

```
以下は `/hookify` で自動化する価値がありそうです (今回は保存のみ行いました):
- [項目] — 検知トリガー案: [PreToolUse on Bash if command matches X]
- ...

自動化したければ `/hookify` を別途起動してください。
```

**このスキル内で `/hookify` を起動してはならない**。案内にとどめる。自動化は別の意思決定なので、ユーザーの明示的なアクションに委ねる。

向く例: 「コミット前に lint/test を流すのを忘れた」「特定のディレクトリで作業する前に CLAUDE.local.md を読み忘れた」のような、**ツール実行や発話のパターンで機械的に検知できるもの**。

向かない例: 「相手の意図を先回りして読み違える」のような、検知ロジックを書きにくい抽象的な反省。これは memory/CLAUDE.md で抑える。

## やらないこと

- 良かった点・承認されたアプローチのふりかえり (**このスキルは反省点に特化**)
- Claude からユーザーへの要望出し (対象外)
- 自動で保存 (毎項目ユーザーの許可を取る)
- `/hookify` を自分から起動 (候補紹介のみ)
- 抽象的な感想 ("もっと丁寧に進めるべきでした" のような中身の無い反省) を書く
- 気付きが無いのに無理に埋める (「特筆すべき反省点はありません」と素直に言う)
- **権限拒否ログを Claude が判定する** (許可すべき / 拒否で正しい の分類はしない。一覧出力で終わり、判定はユーザー)
- **新規スキルをこのスキル内で作る** (候補提示まで。作成は `/skill-creator` の仕事)
- **CLAUDE.md をドキュメント更新候補の対象にする** (CLAUDE.md は §3 の他カテゴリで扱う。ここでの対象は `docs/`, `README.md`, 各 `SKILL.md` などの非 CLAUDE.md)

## 省略時の挙動

- 会話が短く・反省点が無い場合: カテゴリのどれにも該当なしと伝えて終了
- ユーザーが「Claude の提示は要らない、私の言う反省点だけ保存して」と指示した場合: 抽出フェーズをスキップしてユーザー入力のみ処理する
- 会話履歴が圧縮されて拾いきれない場合: `~/.claude/projects/<project-dir>/<session-id>.jsonl` から transcript を直接読む

## 例

新規保存ケース (既存ルールに同趣旨なし):

> 書き物の章立てを合意せず本文を書き切り再作業 (2 回) / 根本原因: 構造を詰めずに着手 / 推薦: `~/.claude/CLAUDE.md` / スコープ: ユーザー全体 / 強度: 常時

既存ルール該当ケース (同趣旨ルールが既存):

> CLAUDE.md の内容を確認せず断言 / 根本原因: 印象で答えた / 既存ルール: 「知らないこと・曖昧なことは勘で埋めずに〜」 / 改善案: なし (既存で覆えている)
