# Session Feedback — 反省・改善ふりかえりスキル

セッション中に Claude が犯した失敗や、もっと効率よくできたはずのやりとりを洗い出し、**保存先まで選び切ってから**永続化レイヤに記録する。

## ワークフロー

```
[1. 抽出] 機械抽出 → 主観抽出 (問題のあるものだけ)
   ↓
[2. 一覧提示] 全カテゴリを Markdown テーブルで提示 + 表外補足
   ↓ 「対応する番号を選んでください (例: 1,3,5)」
   ↓ ユーザが対応可否を一括判断
   ↓
[3. 深掘りループ] 選ばれた番号ごとに
   ├─ メイン表 (該当箇所・詳細・既存ルール照合)
   ├─ 対応案表 (5 列固定、最低 2 案)
   ├─ 差分補足表 (必要なら)
   └─ 採用案を選ばせて書き込み + 進捗表示
   ↓
[終了]
```

## 1. 抽出

### 1.1 機械抽出 (必ず最初にやる)

会話の主観だけだと見落としや圧縮による欠損があるため、付属スクリプトで先に事実を集める。

```bash
session-feedback-extract
```

得られるもの:
- session span (ユーザー turn 数 / assistant turn 数 / 期間)
- 発動した Skill 一覧 (`Skill` ツール経由のみ。SessionStart で auto-load される `using-superpowers` 等は含まれない)
- ユーザーが拒否した tool_use 一覧 ("doesn't want to proceed" シグナルで検出。ask→allow の記録は jsonl に残らないので拾えない。**AskUserQuestion 拒否は除外し下記の別セクションで surface**)
- AskUserQuestion 拒否一覧 (拒否時の元質問文 / header / 選択肢を verbatim で surface。拒否 ≠ 否定で、前提条件不足のシグナルとして扱うため別建てにしている)
- 構造化質問の応答一覧 (AskUserQuestion / auq-web の回答を verbatim で surface)
- mirugit annotations (`~/.mirugit/review-session.json` の token に対応する `sessions/<token>.json` から `author: "user"` のコメントを surface。reply の場合は親 Claude コメントの先頭 120 字も併記)

このあとの主観抽出 (1.2) は、この出力を必ず参照しながら進める。

### 1.2 主観抽出 — 問題のあるものだけ拾う

会話履歴を読み、以下のカテゴリで具体的な改善点を集める。**問題のないものは載せない**。「適切に機能したスキル」「整合済みのドキュメント」などは §2 表に出さない。**該当無しのカテゴリは表を出さず、§2 末尾の `提案なし:` 行にカテゴリ名を並べる**（全カテゴリをチェックした事実をユーザに見せるため、省略はしない）。

| カテゴリ | 拾う対象 |
|---------|---------|
| 繰り返した失敗 | 同じ性質のミスを 2 回以上やった瞬間 |
| 不要な確認 | ユーザが**回答した**質問 (AUQ / 自然文問わず) のうち、手元の情報で解決できたもの |
| 早すぎる質問 | 1.1 の AUQ 拒否一覧のうち前提情報不足が原因のもの (a: 質問不要 / b: 情報不足) |
| 使用スキル分析 | トリガー漏れ / description 改善余地 / 反復した手作業 (新規候補) |
| ドキュメント更新候補 | コード/挙動と `docs/`・`README.md`・各 `SKILL.md` 等の不整合 (CLAUDE.md は他カテゴリで扱うので対象外) |
| 権限見直し | 1.1 の denied 一覧 (AskUserQuestion 拒否を除く) を Claude が読み、`allow へ格上げ` / `条件付き allow` / `拒否継続` 等の案を提示 |
| その他の改善点 | ユーザから修正が入った行動、遠回りな手順、非効率な判断 (上記カテゴリに当てはまらないもの) |
| hookify 候補 (参考) | ツール実行・発話のパターンで機械的に検知できる失敗 |

抽出のコツ:

- 各項目について「**これを保存しなければ、次回以降 Claude は再び同じミスをするか？**」を問う。No なら記録候補から外す
- 失敗の**症状だけでなく根本原因**も一緒に書く
- 各項目は**既存の永続化先と照合**する: `~/.claude/CLAUDE.md` / `<project>/CLAUDE.md` / `<project>/CLAUDE.local.md` / `.claude/rules/*.md` / auto memory。同趣旨ルールがあれば「既存ルールあり」とタグ付け
- 1.1 の**構造化質問の応答**を読む: 回答中の自由記述・その他記述の修正・注文は「その他の改善点」、手元情報で解けた質問は「不要な確認」へ回す。素の選択（推奨どおり等）だけはシグナルにしない
- 1.1 の **AskUserQuestion 拒否一覧**を読む: 拒否は否定ではなく、**前提情報が不足したまま質問してしまったシグナル**であることが多い。元質問の question / options を見て (a) 手元 (一次情報・git・既存実装・memory) で確定できたなら質問自体が不要だった / (b) ユーザ判断は妥当だが選択肢の判断材料 (現状・トレードオフ・関連パス) が質問文に含まれず答えられなかった、のどちらかなら「早すぎる質問」へ。それ以外 (好み・方針の純粋な選択を保留した) はノイズとしてスキップ
- 1.1 の **mirugit annotations** を読む: file:line に紐づく直接的な訂正シグナル。Claude のレビューアノテーション (`author: "claude"`) への返信 (reply) は内容で振り分ける — 「違う」「直して」「要らない」系の訂正は `その他の改善点` か `繰り返した失敗`、 用語/挙動の質問は `ドキュメント更新候補` 候補。`実行して` `LGTM` 等の素の承認はシグナルにしない
- 特定 SKILL.md の改善を提案する前に必ず Read する。重複・矛盾を防ぐため

## 2. 一覧提示 (1 表に統合)

全カテゴリの項目を **1 つの Markdown テーブル**で提示する。カテゴリ固有の詳細は §3 の深掘り時に出すので、§2 では出さない。表外補足は最低限 (内容セルに入りきらない引用が必要なときだけ)。

### 表構成 (5 列固定)

| 列 | 内容 |
|---|------|
| `#` | 通し番号 `#1, #2, ...` (対応対象カテゴリ)。hookify 候補は別系統 `H1, H2, ...` |
| `カテゴリ` | §1.2 のカテゴリ名 (`繰り返した失敗` / `不要な確認` / `早すぎる質問` / `使用スキル分析` / `ドキュメント更新候補` / `権限見直し` / `その他の改善点` / `hookify 候補`) |
| `内容` | このセッションの出来事 (shorthand 50 字以内、書き方は表記の約束) |
| `推薦先` | 短縮記法 (`~/CLAUDE.md` / `<proj>/CLAUDE.md` / `memory:feedback` / `edit:<file>` / `案内のみ` 等) |
| `S/W` | スコープ + 強度の合成 |

S/W 表記:

| 記号 | スコープ | 強度 | 典型 |
|------|---------|------|------|
| `U-常` | User (全環境) | 常時参照 | `~/.claude/CLAUDE.md` |
| `P-常` | Project | 常時参照 | `<proj>/CLAUDE.md` |
| `P-関` | Project | 関連時のみ | `memory:*` (auto memory) |
| `—` | — | — | hookify 候補 / 案内のみ等で該当なし |

### 表記の約束

- セル内は 1 行 (`<br>` / 行末 2 スペースは Claude Code レンダラ非対応 — issue #26390)
- 「内容」は **shorthand** (`A → B → C` / `A した後 B で C` 等) でこのセッションの出来事を 50 字以内。超えたら表外補足へ
- 「内容」では **自明** (AUQ 拒否等) や**他セッションでも書ける一般描写**を避ける (why: 文章化すると一般描写に流れて特異性が薄まる)
- パスは `~` 表記、推薦先は `memory:feedback` / `edit:<file>` 形式、S/W は上表に従う
- カテゴリ固有のフィールド (根本原因 / パターン a/b / 元質問サマリ等) は §2 では出さない。§3.1 で深掘り時に出す
- **表外補足は最低限**。基本省略可。50 字を超える引用や、サマリに収まらない 1 文の補足が必要なときだけ `### 補足` に `N. ...` 形式で書く

### 「提案なし」行

該当が 1 件もないカテゴリは、表の下に 1 行で:

```
提案なし: 不要な確認, ドキュメント更新候補, 権限見直し
```

### 提示後のプロンプト

```
対応する番号を選んでください (例: 1,3,5)
※ H 行は参考情報です。/hookify 反映の判断は別途お願いします
```

ユーザの入力で対応対象が確定。選ばれなかった番号 = スキップ。

## 3. 選ばれた番号ごとの深掘りループ

各 `#N` について以下のフォーマットで提示する。**項目ごとに同じ構造を繰り返す**。

### 3.1 メイン表 (5 行固定)

§2 表のサマリを根拠まで展開する。フィールド名は §2 と一致させる。

| 行 | 条件 | 内容 |
|---|-----|------|
| `#N` | 必須 | サマリ (§2 表と同じ文) |
| 該当箇所 | 必須 | 第 M ターン目 + 状況 1 行 + 会話の引用 |
| 詳細 | 必須 | カテゴリ別フィールド (下表) |
| 既存ルール | 必須 | `なし` / `あり: "ルール文" (出典: <パス>)` |
| 発動しなかった理由 | 既存ルールありのみ | 射程・抽象度・トリガー条件のどれが問題か |

カテゴリ別の「詳細」フィールド:

| カテゴリ | 詳細フィールド |
|---------|--------------|
| 繰り返した失敗 | 根本原因 / 繰り返し回数 |
| 不要な確認 | 参照すべきだった情報源 |
| 早すぎる質問 | パターン (a:質問不要 / b:情報不足) / 元質問サマリ (question / options) / 揃えるべきだった前提 |
| 使用スキル分析 (本文補強) | 対象 SKILL.md / 追記したい文 |
| 使用スキル分析 (desc/新規) | トリガー漏れ状況 / 反復手作業 |
| ドキュメント更新候補 | 不一致箇所の引用 |
| 権限見直し | 拒否された tool + input サマリ / 拒否回数 / 再発予測 |
| その他の改善点 | 改善方向 / 何が遠回りだったか |

### 3.2 対応案表 (5 列固定、最低 2 案)

| 案 | アクション | 対象 | 効果 | 懸念 |
|---|-----------|------|------|------|
| A | 動詞で 1 行 | パス + S/W (or `—`) | 採用するとどう良いか | 副作用・限界 (なければ `—`) |
| B | 別案 | 同様 | 同様 | 同様 |

ルール:

- 列は 5 つ固定 (案 / アクション / 対象 / 効果 / 懸念)
- **最低 2 案**。思考の硬直化防止。3 案以上もよい
- 2 案目が思いつかない場合は「現状維持・skip」「別カテゴリへ移送」「既存ルールへ統合」等を案 B に充てる
- 懸念がなければ `—`。空セルにせず明示する
- 案内のみ (書き込みなし) でも表で出す。表外プレーンテキスト不可

### 3.3 差分補足表 (オプション)

before/after の差分が必要な場合、対応案表の下に置く。見出しは `差分 (案 X)`。

| before | after |
|--------|-------|
| 旧記述 | 新記述 |

### 3.4 採用と書き込み

ユーザが採用案を決定 (A / B / C / カスタム / スキップ)。承認されたら書き込む。

書き込み時の注意:

- memory は `auto memory` システム (グローバル CLAUDE.md 参照) の規約に従う。feedback memory は `**Why:**` `**How to apply:**` を付け、`MEMORY.md` の index 更新も忘れない
- CLAUDE.md は既存の章立てを尊重し、関連セクションに追記する。新しいトップレベル見出しを乱造しない
- 相対日付 (「今週」「Thursday」) は絶対日付に変換してから保存する
- **既存エントリとの重複チェックは書き込み前に必ず行う**。同趣旨のルールがすでにあるなら更新 (追補・具体化) に倒す

カテゴリ別の処理:

| カテゴリ | 書き込み先 / 処理 |
|---------|------------------|
| 繰り返した失敗 / 不要な確認 / 早すぎる質問 / その他 | memory or CLAUDE.md (採用案の `対象` に従う) |
| 既存ルールあり (改善案採用) | 既存ルール本文を Edit |
| 使用スキル分析 (本文補強) | 該当 `SKILL.md` を Edit（dotfiles 管理スキルは除く。下記注記） |
| 使用スキル分析 (desc / 新規候補) | 書き込まず `/skill-creator` 案内のみ |
| ドキュメント更新候補 | 該当ファイルを Read → Edit |
| 権限見直し | `update-config` スキル案内、または該当 `settings.json` を直接 Edit (採用案の `対象` に従う) |

- **dotfiles 管理スキル (`dotfiles/.claude/skills/` 配下) の本文補強は直接 Edit しない**。skill-md-reviewer + sanitize-auditor のゲートを持つ retro → process-retro 経路に回す（単体起動時も同じ。直 Edit はレビュー未通過の変更を公開リポに入れてしまうため）。

進捗表示 (固定):

```
✅ #N 保存完了 (推薦先: <パス>)  進捗: 3/5
```

## 例

### 例 1: 繰り返した失敗 — 既存ルールなし → 新規保存

| 項目 | 内容 |
|------|------|
| **#1** | テーブルのセル内に `<br>` を 3 回書いて表示崩れ |
| **該当箇所** | 第 12 / 18 / 24 ターン目、SKILL.md の §2 を編集中 |
| **詳細** | 根本原因: 表レンダラの制約 (issue #26390) を再確認しなかった / 回数: 3 |
| **既存ルール** | なし |

対応案

| 案 | アクション | 対象 | 効果 | 懸念 |
|---|-----------|------|------|------|
| A | feedback memory に新規追加 | `memory:feedback` (P-関) | 別リポでも表崩れを再発防止、軽量 | 関連作業のときしか参照されない |
| B | `~/.claude/CLAUDE.md` に 1 行追加 | `~/.claude/CLAUDE.md` (U-常) | 毎セッション必ず参照される | 表編集は限定文脈、CLAUDE.md だと過剰 |

### 例 2: 不要な確認 — 既存ルールあり → 改善案 3 案

| 項目 | 内容 |
|------|------|
| **#3** | setup.sh の env 引数を毎回ユーザに聞いていた |
| **該当箇所** | 第 5 ターン目、`"home と work どちらでマージしますか？"` |
| **詳細** | 参照すべきだった情報源: `basename "$(readlink ~/.zshrc.local)" .zsh` |
| **既存ルール** | あり: `"勘で埋めずに、真っ先に一次情報に当たること"` (`~/.claude/CLAUDE.md`) |
| **発動しなかった理由** | 「ユーザ確認より先に手元のシグナルを探す」が射程に入っていない |

対応案

| 案 | アクション | 対象 | 効果 | 懸念 |
|---|-----------|------|------|------|
| A | 「ユーザ確認より先に一次情報」へ書き換え | `~/.claude/CLAUDE.md` L8 | 一般性維持しつつ射程拡張 | 「一次情報」の解釈が広い |
| B | 確認先を列挙する形に書き換え | 同上 | 具体的で行動に直結 | 列挙が古びる |
| C | 案A本文 + 実例併記 | 同上 | 抽象+具体で再発防止に強い | 実例陳腐化リスク |

### 例 3: 使用スキル分析 — 新規候補 (案内のみ)

| 項目 | 内容 |
|------|------|
| **#7** | kustomization.yml の ref / newTag 書き換え作業を 4 回繰り返した |
| **該当箇所** | 第 8 / 14 / 19 / 27 ターン、dev 環境動作確認用 PR 作成中 |
| **詳細** | 反復手作業: ブランチ名 → ref / SHA → newTag への変換、PR 本文の定型 |
| **既存ルール** | — |

対応案

| 案 | アクション | 対象 | 効果 | 懸念 |
|---|-----------|------|------|------|
| A | `/skill-creator` を別途起動して新規スキル化 | — | eval 検証つきで description 精度が高い | 別セッションが必要 |
| B | 既存スキルへの統合可能性を検討 | 既存スキル | スキル増殖を抑える | 文脈が異なる場合は統合困難 |

## 保存先の対応表

| 場所 | スコープ | 強度 | type (memory のとき) |
|------|---------|------|---------------------|
| `~/.claude/CLAUDE.md` | ユーザー全体 | 常時 | — |
| `<proj>/CLAUDE.md` | プロジェクト (共有) | 常時 | — |
| `<proj>/CLAUDE.local.md` | プロジェクト (ローカル) | 常時 | — |
| `memory:feedback` | プロジェクト | 関連時 | feedback (行動ルール) |
| `memory:user` | プロジェクト | 関連時 | user (属性) |
| `memory:project` | プロジェクト | 関連時 | project (作業状況) |
| `memory:reference` | プロジェクト | 関連時 | reference (外部ポインタ) |
| `edit:<file>` | — | — | ドキュメント / SKILL.md 直接編集 |
| `案内のみ` | — | — | `/skill-creator` `/update-config` `/hookify` 等への案内、書き込みなし |

迷ったら:
- スコープが迷ったら **ユーザーに確認する**
- 強度が迷ったら **memory** を優先 (CLAUDE.md を肥大化させない)

## やらないこと

このスキルでやってしまいがちな反転ルール:

- 良かった点・承認されたアプローチのふりかえり (このスキルは反省点に特化)
- Claude からユーザーへの要望出し (対象外)
- 抽象的な感想 ("もっと丁寧に進めるべきでした" のような中身の無い反省) を書く
- 気付きが無いのに無理に埋める (「特筆すべき反省点はありません」と素直に言う)
- `/hookify` や `/skill-creator` を自分から起動する (候補は §2 / §3 で提示するだけ。自動化や新規スキル作成は別の意思決定)

## 省略時の挙動

- 会話が短く・反省点が無い場合: カテゴリのどれにも該当なしと伝えて終了
- ユーザーが「Claude の提示は要らない、私の言う反省点だけ保存して」と指示した場合: 抽出フェーズをスキップしてユーザー入力のみ処理する
- 会話履歴が圧縮されて拾いきれない場合: `~/.claude/projects/<project-dir>/<session-id>.jsonl` から transcript を直接読む
- ユーザが §2 で番号を 1 つも選ばなかった場合: §3 ループをスキップし「対応する項目がありませんでした」と伝えて終了
- §2 の番号選択待ち中に別スキルが呼ばれた場合: session-feedback の状態を破棄し、別スキルを優先する
