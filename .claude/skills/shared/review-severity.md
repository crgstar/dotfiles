# レビュー重要度の定義

| 重要度 | 基準 | 例 |
|--------|------|-----|
| 高 | 本番環境で不具合・データ損失が発生する、またはセキュリティリスクがある | 未処理の例外、データ競合、XSS |
| 中 | 現在のコード・設定で実際にUX劣化やバグが発生する条件が存在する | 特定の操作で表示崩れ、エラー時にフィードバックなし |
| 低 | 上記に該当しないすべての指摘 | パターン不整合、テスト不足、将来の保守リスク、命名の改善 |

## 判断原則

- **実害ベースで判定する。** 「将来問題になるかもしれない」「パターンが不統一」は低。現時点で問題が発生する条件が存在して初めて中以上
- **「高」はブロッキングな問題のみ。** マージを止めるべきレベル
- **迷ったら低にする。** 重要度のインフレは指摘の信頼性を下げる
- **昇格ルール:** 具体的な理由を明記すれば、基準より1段階だけ上げてよい（例: 低→中）。理由なしの昇格は禁止
