## 基本原則

初見の Claude Code の判断・行動を変えない行は削る。
長く正確に書くよりも、短く印象に残る書き方をする。多少抽象的でも、長文は読み流されて行動を変えないため。

ルール・表・チェックリストを書きたくなったら一旦止めて、スタンス 1 文にできないか試す。
迷ったら短い方を選ぶ。

ただし「印象に残る」 を警句や煽り文句で達成しない。抽象ルールには why を 1 文添える。
why は判断を変える根拠 1 文に限る。経緯・代替案・トレードオフは実行に不要なので書かない（git log / memory へ）。

依存スキル・CLI の仕様は写経しない。「実行は X に従う」と書き、自分側固有の差分だけ残す。
依存先が更新されたとき drift するため。連携を書く前に依存先を動かして挙動を確かめる。

環境固有の識別子（メールアドレス・絶対パス・組織名等）は SKILL.md に含めない。public リポに入ると取り消せないため。

## description は 3 要素

対象物 + 発動条件 + 除外条件 を揃える。動詞中心ではなく対象物で語ると誤発動が減る。
圧縮の原則は本文向け。発動可否は description の語マッチで決まるため、トリガー語の列挙は削らない。

## 書いたら subagent でレビュー

収束を確認してから完了と報告する。

`skill-md-reviewer` サブエージェント（`subagent_type: skill-md-reviewer`）に、レビュー対象 SKILL.md と
この SKILL.md のフルパスを渡してレビューを依頼する（基準は本ガイドを Read させる。写経しない）。
レビュー対象は今回変更した箇所に限る。pre-existing 行への指摘はスコープ外として却下する。
妥当な指摘だけ反映し、却下する指摘は理由を次ラウンドで reviewer に渡す。
再レビューは最大 3 回。新規指摘が 0 件なら収束。3 周で収まらなければ残件をリストでユーザに渡す。
