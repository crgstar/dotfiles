# テスト設計ガイド

## 命名
- it の説明文はユーザー視点・ドメイン用語で日本語で書く。実装詳細や内部変数名を漏らさない
- テスト名は全ての assert を正確に説明する。検証対象外のものが名前に含まれていたり、除外条件が省略されていないか確認する
- NG: `0件時はdisabledになる` / OK: `商品が0件の場合は購入ボタンを押せない`

## describe 構造
- 第1階層: 関数名 or コンポーネント名
- 第2階層: シナリオの特徴
- it: 期待される振る舞いを書く（入力の特徴ではなく）

## 対称性テスト
- 対になるものは、片側を書いたら常にもう片側との対称性を確認する
- 逆関数ペア（例: シリアライズ / デシリアライズ）: 独立した2つの変換。「ラウンドトリップ」と呼ばない。テストデータを共有し、describe で「〇〇と△△の対称性」、it で方向を明示
- 類似機能ペア（例: 作成ダイアログと編集ダイアログ）: 一方を追加・変更したら常にもう一方と観点を突き合わせる。揃える粒度は実装で挙動が自明かによる — 共通実装で自明な観点は片側で足り、種別ごとに分岐する非自明な観点（初期値・固有フィールド・エラー等）は両側に書く

## AAA (Arrange-Act-Assert)
- 空行で3セクションを区切る。コメントは不要
- **Guard Assertion**: 「Xの場合にdisabledに*なる*」をテストするなら、Actの前にenabledであることをassertする。Act前後で状態が変化することを明示しないと、別の理由で最初からdisabledだった場合にテストが意味をなさない
- **Arrangeの十分性**: Actがテスト対象の唯一の変化要因になるようArrangeを構築する。複数の条件が同時に結果を引き起こす状態でassertしても、どの条件をテストしているか分離できない

## テストの独立性
- 各テストは他テストの結果に依存しない。共有状態は beforeEach でリセット

## 境界値
- 空配列・空文字列・null・undefined・0・1・最大値・off-by-one を意識

## テストデータ
- 多数のフィールドを持つオブジェクトが入力の場合、ファクトリ関数でデフォルト値を用意しテストごとの差分だけ明示する（例: `createUser({ role: 'admin' })`）
- マジックナンバーを避け意図が伝わる値を使う

## TDD
1. RED → 2. GREEN → 3. REFACTOR (/simplify。一時的に独立させた describe の統合もここで行う)

## モック
- 外部依存のみモック。内部実装はモックしない
- モックしすぎは実装詳細への結合 → テストが壊れやすくなる兆候

## 冗長テストの判断
- 仕様の網羅性が最優先、DRY はその次
- 観察可能なふるまい（API 呼び出し・通知・UI 状態）は境界値ごとに独立して維持。実装を根拠に「結果が同じだから削る」はホワイトボックス的削減で、仕様変更の regression を見逃す
- 削除判断はユーザーに確認

## 類似機能間の網羅性チェック
- 類似機能を追加したとき、既存の describe ブロックのテスト一覧と突き合わせて観点の漏れがないか確認する

## 既存テストとの整合性
- 既存に大きな欠陥（観点漏れ・不正モック等）がなければ、describe 階層・命名・モック方針は既存に揃える
- 新規テストは末尾に新 describe を足す前に、既存の関連 describe 内（関連テストの直後）に置けないか確認する
- it 文は語彙だけでなく文型も同 describe の既存テストに揃える — 文型の乱れは describe 内のスキャンを遅くし、レビューの見落としを生む
- 欠陥が本質的なら別課題として切り出し、スタイル刷新とは混ぜない

## 重要度
- レビュー時の重要度判定は `.claude/skills/shared/review-severity.md` の定義に従う
