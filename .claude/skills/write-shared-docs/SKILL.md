# 書く前に 1 つだけ決める

**誰が読むのか・読者はそこから何を持って帰るのか** を 1 行で言語化してから書き始める。言語化できないなら先に聞く。
why: 決まっていないと、書き手の知識を並べる文書になる。

この問いへの答えは作業前確認用であり、文書本文には書かない。

# 最新の確定結果だけを書く

検討の途中経過・没案・草案・経過報告は混ぜない。
why: 草案が混ざると読者は「どれが現行仕様か」を本文から逆算する羽目になる。試行錯誤の置き場は ADR / issue。

# 議論していない決定を書かない

ユーザと合意していない技術選択 (権限・上限値・タイムアウト・実装手段・レイヤー配置・テスト方針など) は、書く前に「ここまで含めますか?」と聞く。
why: 書き手の補完は、読者には「合意済みの仕様」として届く。

# 自分が調べた過程を載せない

「最初は X を検討した」「Y のドキュメントによると」のような調査ログは消す。
why: 書き手の理解プロセスは、読者には「本筋に辿り着くまでのノイズ」として読まれる。

# 汎用セクションを反射で足さない

「実装メモ」「テスト方針」「将来の検討余地」「補足情報」は、読者が必要としない限り書かない。迷ったらユーザに聞く。
コードの行番号・スニペット・実装箇所も同様。読者が開発者であっても、実装仕様書でなければ書かない。
why: 反射で並べた章は中身が薄くなり、読者は「ここは読まなくていい」と全体の信頼度を下げる。

# 記号ラベルは文書内で定義する

会話でだけ通じるラベル (「案 A」「パターン X」等) は定義なしで本文に持ち込まない。
why: 記号は書き手 (と議論を共にした人) にしか通じない。
